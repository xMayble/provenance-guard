# Provenance Guard

A backend a creative-sharing platform could plug into to check whether a piece of text was
written by a person or generated by AI. It doesn't just give a yes/no. It returns a
confidence score, a plain-language label for readers, keeps an audit log of every decision,
and lets creators appeal a result they think is wrong.

The whole thing is built around one idea: **wrongly calling a real person's writing "AI" is
worse than missing some AI text.** So the thresholds, the label wording, and the appeals
path are all tilted toward being careful about false accusations.

## Setup

```bash
python -m venv .venv
source .venv/Scripts/activate      # Windows (Git Bash);  use .venv\Scripts\activate on cmd
pip install -r requirements.txt
```

Create a `.env` file in the repo root (it's gitignored, never commit it):

```
GROQ_API_KEY=your_key_here
```

Run the server:

```bash
python app.py
```

It serves on `http://localhost:5000`.

## Endpoints

| Method | Path | Body | What it does |
|---|---|---|---|
| POST | `/submit` | `{text, creator_id}` | Classify text, return attribution + confidence + label |
| POST | `/appeal` | `{content_id, creator_reasoning}` | Contest a result; sets status to `under_review` |
| GET  | `/log` | — | View the audit log (newest first) |

Example:

```bash
curl -s -X POST http://localhost:5000/submit \
  -H "Content-Type: application/json" \
  -d '{"text": "The sun dipped below the horizon...", "creator_id": "test-user-1"}'
```

## Try it yourself (no curl needed)

Two helper scripts talk to the running server, so you don't need `curl`. With the server
running (`python app.py`), open a second terminal and run either:

- **`python try_it.py`** — interactive: type or paste any text, press Enter, and see the
  attribution, confidence, both signal scores, and the reader label. Type `quit` to exit.
- **`python demo.py`** — a scripted end-to-end tour: submits a human-sounding and an
  AI-sounding text, files an appeal, and prints the audit log.

## Architecture overview

The path a submission takes from input to label:

```
POST /submit {text, creator_id}
  -> rate limiter checks the caller
  -> Signal 1: llm_signal()   asks Groq how AI-like the text reads   -> llm_score (0-1)
  -> Signal 2: stylometry()   measures the shape of the text         -> style_score (0-1)
  -> combine()    weighted average (0.65 llm + 0.35 style)           -> p_ai (confidence)
  -> classify()   p_ai -> likely_ai / uncertain / likely_human
  -> make_label() p_ai -> reader-facing label text
  -> store.record_submission()  writes the decision to the audit log
  -> JSON response { content_id, attribution, confidence, label, signals, status, timestamp }
```

The appeal flow looks up the original record by `content_id`, flips its status to
`under_review`, and logs the creator's reasoning next to the original decision so a human can
review it via `GET /log`. (Full diagrams are in [planning.md](planning.md).)

Code is split into three files: `detection.py` (the two signals + scoring + labels),
`store.py` (the JSON-file audit log and content store), and `app.py` (the Flask endpoints).

## Detection signals

Two signals that look at genuinely different things — one reads meaning, one measures shape.

### Signal 1: LLM semantic judgment (Groq, `llama-3.3-70b-versatile`)

- **Measures:** holistically, does the meaning/voice/tone read as AI-generated? The model
  returns a probability (0-1) plus a one-line reason.
- **Why I chose it:** LLMs are good at recognizing the generic, hedged, formulaic prose they
  tend to produce themselves ("It is important to note that...", "Furthermore..."). No simple
  math captures tone like this.
- **What it misses:** these detectors aren't reliable ground truth. They over-flag formal or
  non-native English writing, and the score wobbles a little between calls. I treat it as one
  strong opinion, not the truth, which is why a second independent signal matters.

### Signal 2: stylometric heuristics (pure Python)

Measures the statistical shape of the writing, independent of meaning. I compute several
metrics but only **two feed the score**:

1. **Burstiness** — standard deviation of sentence length (in words). Humans mix long and
   short sentences; AI tends to keep them uniform. Low variation looks more AI.
2. **Informality** — casual markers per 100 words (contractions, lowercase sentence starts,
   `...`, `!!`, ALL-CAPS emphasis of 3+ letters). Casual human writing is full of these; AI
   text is clean.

`style_score = 0.35 * burstiness + 0.65 * informality` — informality leads because it
separates casual human writing from clean AI more reliably than burstiness does on short
text.

Two more metrics — **type-token ratio** and **mean sentence length** — are computed and
returned for transparency but **deliberately not scored**, because they track formality and
length rather than authorship (a wordy human and a wordy AI both score high). Scoring them
would just manufacture false confidence.

- **What stylometry misses:** it's fooled by uniform-but-human writing — formal/academic
  prose and simple repetitive poetry both look "AI-shaped." That's the known false-positive
  case, and it's exactly why the LLM signal and the appeals path exist.

## Confidence scoring

The two signals are combined into one number with a weighted average:

```
p_ai = 0.65 * llm_score + 0.35 * style_score
```

The LLM leads (0.65) because it reads the whole thing and is the stronger detector.
Stylometry (0.35) is a consistent, deterministic sanity check that doesn't change between
runs, so it backs the LLM up rather than overruling it. That `p_ai` is the **confidence
score: my estimate of the probability the text is AI-generated**, from 0 to 1. `0.5` means
"genuinely can't tell"; the further from `0.5`, the more sure.

The score maps to three bands. The bands are **asymmetric on purpose** — the bar to call
something AI is high and the uncertain middle is wide, so borderline work is called
"uncertain," not falsely accused:

| `p_ai` | attribution | label |
|---|---|---|
| `>= 0.70` | `likely_ai` | high-confidence AI |
| `0.45 – 0.70` | `uncertain` | uncertain |
| `< 0.45` | `likely_human` | high-confidence human |

So a 0.51 produces an *uncertain* label and a 0.95 a *strong AI* label — there's no binary
flip at 0.5.

### How I validated it's meaningful

I ran four deliberately chosen texts (clearly AI, clearly human, formal human, lightly-edited
AI) straight through the pipeline and checked the scores spread out into sensible, separated
bands — and that the two signals were printed separately so I could see which one drove each
result. The clearly-AI and clearly-human cases land far apart, and the formal-human case
(the classic false-positive trap) stays in `uncertain` instead of being accused. Results were
identical across repeated runs, so the scoring is stable.

### Two real examples (noticeably different confidence)

Captured live from `/submit`:

**High-confidence case — clearly AI text** (`confidence = 0.87`):
> "Artificial intelligence represents a transformative paradigm shift in modern society. It is
> important to note that while the benefits are numerous... Furthermore, stakeholders must
> collaborate to ensure responsible deployment."
>
> `llm_score = 0.90`, `style_score = 0.83` → `p_ai = 0.87` → **likely_ai**

**Lower-confidence case — short personal note** (`confidence = 0.35`):
> "The sun dipped below the horizon, painting the sky in hues of amber and rose. I sat on the
> porch, coffee in hand, watching the neighborhood slowly go quiet."
>
> `llm_score = 0.10`, `style_score = 0.83` (short text, burstiness flagged unreliable) →
> `p_ai = 0.35` → **likely_human**

The two signals disagree on the second one (stylometry sees clean uniform text and leans AI;
the LLM hears a personal human voice and leans human). The LLM's higher weight wins, which is
the intended behavior — and a good illustration of why one signal alone isn't enough.

## Transparency label

The label shown to a reader changes based on the confidence band. `{pct}` is a percentage
derived from the score. The three variants, verbatim:

**High-confidence AI** (`p_ai >= 0.70`; `{pct}` = `round(p_ai*100)`):
> 🤖 Likely AI-generated. Our automated analysis suggests this content was probably created
> with significant AI assistance (estimated {pct}% likelihood). This is an automated estimate,
> not a verdict, and detection isn't perfect. If you're the creator and think this is wrong,
> you can appeal and a human will review it.

**High-confidence human** (`p_ai < 0.45`; `{pct}` = `round((1-p_ai)*100)`):
> ✍️ Likely human-written. Our automated analysis didn't find strong signs of AI generation
> (estimated {pct}% likelihood this is human-written). This is a probability, shown just for
> context.

**Uncertain** (`0.45 <= p_ai < 0.70`; `{pct}` = `round(p_ai*100)`):
> ❓ Uncertain. We couldn't confidently tell whether this is human- or AI-written (estimated
> {pct}% AI-likelihood). Rather than guess, we're marking it inconclusive.

The AI label is intentionally softened — it never flatly accuses, it states it's an estimate,
and it points to the appeal path — because a false positive is the costly mistake here.

## Appeals workflow

A creator who disagrees with a classification POSTs to `/appeal` with the `content_id` and
their `creator_reasoning`. The system then:

1. Looks up the content by id (returns 404 if unknown).
2. Changes its status from `classified` to `under_review`.
3. Appends an appeal entry to the audit log, kept next to a snapshot of the original decision
   (attribution, confidence, both signal scores) plus the reasoning and a timestamp.
4. Returns a confirmation.

Automated re-classification is **not** done — appeals route to a human. A reviewer reading
`GET /log` sees each `under_review` item with its original scores and the creator's reasoning
right there, which is enough to make a call.

```bash
curl -s -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "PASTE-ID-HERE", "creator_reasoning": "I wrote this myself..."}'
```

## Rate limiting

Applied to `/submit` with Flask-Limiter, per client IP:

```
@limiter.limit("10 per minute;100 per day")
```

**Reasoning for these numbers:** a real creator submits their own work occasionally — maybe a
handful of pieces in a sitting, sometimes re-checking after an edit. **10 per minute** is
generous for that legitimate burst but stops a script from hammering the endpoint (which also
matters because every submission costs a Groq API call). **100 per day** caps sustained abuse
from a single IP while still allowing an unusually heavy but real day of work. The limits are
per-IP, so one bad actor can't starve everyone else.

**Evidence** — sending 12 rapid requests (I'd already made 2 submissions in that minute, so 8
more reached the limit of 10, then the rest were blocked):

```
request 1 -> HTTP 200
...
request 8 -> HTTP 200
request 9 -> HTTP 429
request 10 -> HTTP 429
request 11 -> HTTP 429
request 12 -> HTTP 429
```

Blocked requests get a clean JSON `429` response, not an HTML error page.

## Known limitations

These are cases the system will get wrong, tied to how the signals work:

1. **Formal or non-native (ESL) human writing.** Polished academic prose is even,
   contraction-free, and a little formulaic, so both signals drift toward "AI." It's the
   classic false positive. My defenses are the high AI threshold (so it usually lands in
   `uncertain`, not `likely_ai`), the softened label, and the appeal path — but it can still
   be misjudged.
2. **Simple, repetitive poetry.** Refrains and short, even lines mean low burstiness and few
   casual markers, so stylometry reads it as AI even though the uniformity is a deliberate
   human device.
3. **Lightly-edited AI text.** Text that's AI-generated but reworded to add a personal voice
   reads human to both signals and can land in `likely_human` (I saw this in testing — it
   scored 0.19). This is a false negative, which by our design is the *less* costly error than
   falsely accusing a human, but it's a real blind spot worth naming.

## Spec reflection

- **One way the spec helped:** writing `planning.md` first forced me to pin down the exact
  thresholds and label wording before any code existed. So when my first calibration came out
  wrong (formal human was being labeled `likely_ai`), I already had a clear target — "this
  case should land in *uncertain*" — to tune toward, instead of guessing what "good" looked
  like after the fact.
- **One way I diverged:** my plan first set the stylometry weights at `0.6 * burstiness +
  0.4 * informality`. During calibration I found burstiness was too noisy on short text and
  barely separated the test cases, while informality cleanly did. I flipped the weights to
  `0.35 / 0.65` and updated `planning.md` to match, so the doc and code stay in sync.

## AI usage

I used an AI coding assistant, giving it the relevant `planning.md` sections and reviewing /
correcting the output rather than pasting it blindly. Two specific instances:

1. **Stylometry scoring.** I directed it to implement the burstiness + informality metrics and
   a combined score. Testing revealed two bugs in the first version: the ALL-CAPS detector was
   counting the acronym "AI" as informal emphasis (making AI text look human), and burstiness
   was over-weighted so a clearly-AI paragraph scored as `uncertain`. I overrode it: I
   restricted ALL-CAPS to 3+ letters and rebalanced the weights toward informality (0.35/0.65).
2. **The LLM detector prompt.** The first prompt I had it write was too vague — it rated both a
   clearly-AI paragraph and a substantive formal-human paragraph at a lazy 0.70. I rewrote the
   prompt with an explicit 0-1 scoring rubric that tells the model to separate *generic hedged
   filler* from *specific, substantive formal writing*, and to avoid penalizing non-native
   writers. That pushed the clearly-AI case up to ~0.90 and let the formal-human case settle
   into `uncertain`.
